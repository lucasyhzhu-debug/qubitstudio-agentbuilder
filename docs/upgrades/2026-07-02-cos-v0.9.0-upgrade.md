## v0.8.1 — drain fixes (PR #28)

### 1. Kanban state mirroring on every lifecycle move

**Where:** `skills/drain/SKILL.md` (Steps 2, 3, 3a, 4), `skills/drain/references/linear-api.md`.

**Why:** the drain tracked lifecycle purely via labels (`needs-agent` / `needs-lucas` / `done`),
so every issue sat in the Linear board's **Backlog** column forever — the board told the owner
nothing about what the agent was doing.

**Spec requirements:**

1. Every Linear mutation that creates an issue or changes a lifecycle label MUST also set the
   matching workflow `stateId` **in the same mutation** (never a second call):
   | Lifecycle moment | Label state | Kanban state |
   | --- | --- | --- |
   | Triage creates the issue (queued for agent) | `needs-agent` | **Todo** |
   | Step 4 starts acting on an issue | `needs-agent` | **In Progress** |
   | Parked for the owner (proposal or blocker) | `needs-lucas` | **In Review** |
   | Re-armed (thread reply, or Step 3a confirm flip) | `needs-agent` | **Todo** |
   | Completed | `done` | **Done** |
2. Step 4 sets **In Progress** immediately after fetching issue detail, before acting.
3. Every re-park path in the calendar-write pass (draft-missing blocker, multi-block blocker,
   failed provenance reconciliation) sets **In Review** alongside the `needs-lucas` flip.
4. The crash-critical ordering of the calendar write is unchanged: record `event_id` in the
   `## Draft` block FIRST, then post links, and only THEN flip `done` + **Done** + remove
   `confirmed-by-agent` in a single `issueUpdate`.
5. `references/linear-api.md` gains a workflow-states section: the lifecycle↔state table, the
   combined mutation shape `issueUpdate(id, input:{ labelIds:[…], stateId:"…" })`, and a note
   that state ids can be re-fetched with
   `{ team(id:"…"){ states{ nodes{ id name type } } } }` if a mutation returns a state error.

**Scrub note:** workflow state ids are **workspace-specific UUIDs**. Do NOT copy upstream ids.
On first application, fetch the local workspace's state ids with the query above (team id =
`{{LINEAR_TEAM_ID}}`) and record them in `references/linear-api.md` next to the table.

**Acceptance:** every `issueUpdate` in `skills/drain/SKILL.md` that swaps a lifecycle label
mentions `stateId`; `linear-api.md` contains the states table with locally-fetched ids; a live
drain cycle moves a new issue Todo → In Progress → (In Review | Done) on the board.

### 2. Drain precheck: thread-reply detection

**Where:** `scripts/drain-precheck.ps1`.

**Why:** confirms and edits ("confirm 1", scheduling tweaks, follow-ups) arrive as **Discord
thread replies** on parked issues. They advance no channel watermark, and a parked issue is
`needs-lucas` (not `needs-agent`), so neither existing precheck (channel messages, Linear
`needs-agent` query) sees them — a lone confirm sat unprocessed until unrelated work happened to
wake a cycle.

**Spec requirements:**

1. Add a step **4b** to the precheck, running only when steps 3–4 found no work
   (`-not $work`): for each issue tracked in `drain-state.json`'s `issues` map that has a
   `threadId`, fetch up to 5 thread messages after that issue's `lastSeen` marker.
2. Only **non-bot** messages (`author.bot` false) count as work — the drain's own proposal/ack
   posts land after `lastSeen` and must not wake the next cycle in a self-sustaining loop.
3. A 404 on a thread fetch (archived/deleted thread) is caught and skipped — it must not stall
   the whole precheck.
4. Short-circuit: stop fetching as soon as any thread yields work.
5. The Discord REST headers (`Authorization: Bot …` plus the mandatory
   `User-Agent: DiscordBot (<repo-url>, <version>)` — Cloudflare 403s PowerShell's default UA)
   are hoisted above step 3 so both the channel check and 4b share them.

**Acceptance:** with all channel watermarks current and zero `needs-agent` issues, posting a
plain user reply into a parked issue's thread makes the next precheck exit "work exists"; a
bot-authored reply alone does not.

### 3. Env-var case rule (Google auth)

**Where:** `references/google-auth.md`, credentials section.

**Why (real incident):** a var set as `GOOGLE_EMAIL_work` (lowercase suffix) passes PowerShell
smoke tests (case-insensitive) but is **invisible to skills reading env from bash**
(case-sensitive — `$GOOGLE_EMAIL_WORK` → empty). The account looks "not set" headless while
smoking fine.

**Spec requirements:**

1. Document that `<LABEL>` suffixes are **UPPERCASE in every env-var name** even though
   `GOOGLE_ACCOUNTS` lists labels in lowercase.
2. Include the diagnosis (`reg query HKCU\Environment` to check registry name case) and the fix
   (`reg delete` the lowercase value, then `setx` the uppercase one — `setx` alone preserves the
   existing name's case).

**Acceptance:** `references/google-auth.md` states the uppercase rule, the bash-case-sensitivity
reason, and the reg-delete + setx fix.

## v0.8.2 — drain anti-stale-context hardening (PR #29)

**Why (live incident):** with an account already verified connected, a fresh drain cycle told
the owner the opposite — a fabricated env check plus a blocker resurrected from the issue
description's original (stale) proposed action. Root cause: each headless cycle has no memory,
the Linear thread IS its memory, and the description outranked newer verified comments.

### 4. Drain freshness rule / verify-before-assert

**Where:** `skills/drain/SKILL.md`, Step 4.2 (determine the nature of the request).

**Spec requirements:**

1. The newest **verified** statement wins: a drain comment recording a check actually run (token
   mint, API read, env read) supersedes the issue description's original "proposed action" and
   every older comment. The description records what was ASKED at triage time, not current truth.
2. **Verify-before-assert:** never state an account/env/connection/capability status in a
   comment unless the check ran IN THIS RUN. Repeating a status from the description or an older
   comment is how stale blockers resurrect and contradict what the owner was already told.

### 5. Auth-contract precedence

**Where:** `references/google-auth.md`, `GOOGLE_AUTH_KIND_<LABEL>` line.

**Spec requirements:**

1. Check `GOOGLE_AUTH_KIND_<LABEL>` FIRST when assessing whether an account is connected — for a
   `service_account` label, `GOOGLE_REFRESH_TOKEN_<LABEL>` is *expected to be absent* and its
   absence is NOT a blocker.
2. Connection status is only ever diagnosed by minting a token (Step 0) and making one read call
   — never from prior issue comments or thread history (they go stale after env fixes).

**Acceptance (both):** the two rules appear verbatim-in-spirit in their files; a drain cycle on
an issue with a stale "not connected" comment re-verifies before repeating it.

## v0.9.0 — drain conversational-ticket context (PR #30)

The headline upgrade: the drain treats an `#inbox` dictation burst as **one accreting
conversation** instead of a stream of independent messages.

### 6. Adaptive settle window (new files)

**Where:** NEW `skills/drain/references/settle-window.md`, NEW `scripts/settle-lib.ps1`, NEW
`scripts/tests/settle.Tests.ps1`; consumed by `scripts/drain-precheck.ps1` and
`skills/drain/SKILL.md` Steps 2, 3, 4.

**Spec requirements:**

1. A **source** = an `#inbox` channel or an issue's Discord thread. The drain processes a source
   only once it is **settled** — quiet long enough that a multi-message burst is complete.
2. The rule is count-based and **non-bot-scoped** (the drain's own ack/proposal/result posts
   never count and never reset the clock). Considering only non-bot messages newer than the
   source's watermark:
   - 0 non-bot messages → not settled (nothing to do).
   - 1 non-bot message → settled when the newest non-bot message's age ≥ **30s**.
   - 2+ non-bot messages → settled when the newest non-bot message's age ≥ **90s**.
   - **Max-defer ceiling:** if the *oldest* un-processed non-bot message's age ≥ **600s**, the
     source is settled regardless — a perpetually active thread cannot starve.
3. Message age derives from the Discord snowflake id:
   `createdMs = ([long]id >> 22) + 1420070400000` — parse as **Int64** (snowflakes exceed Int32;
   PowerShell 5.1 shifts `[long]`). "Newest" = largest id; "oldest un-processed" = smallest id
   after the watermark. Never rely on array order.
4. One shared implementation (`Test-SourceSettled` in `settle-lib.ps1`) is used by BOTH the
   precheck and the SKILL — never two divergent copies. Constants (30/90/600) are tunable but
   changed in `settle-lib.ps1` and `settle-window.md` together.
5. An un-settled source is skipped with its watermark **un-advanced**, so a later cycle handles
   it once quiet.

**Acceptance:** the Pester suite (`scripts/tests/settle.Tests.ps1`) passes on PowerShell 5.1;
a message posted 5s ago does not get drained; the same source drains on the next cycle after
going quiet.

### 7. Settle gates action, not just ingestion

**Where:** `skills/drain/SKILL.md` Step 4.

**Spec requirements:**

1. Step 4 holds a `needs-agent` issue whose Discord thread has un-settled new non-bot replies —
   it does not act until settled. This closes the mid-dictation double-action hole (acting once
   on a partial burst and again when the rest arrives).
2. The hold is safe for the confirmed-scheduling calendar write too (idempotent via its
   deterministic event id) — no special-casing.

### 8. `#inbox` burst grouping

**Where:** `skills/drain/SKILL.md` Step 2.

**Spec requirements:**

1. A settled channel's new messages are grouped into runs of contiguous **owner-authored**
   messages; each run is handed to `intake` as ONE batch, which may collapse it into a single
   issue (one accreting thought — e.g. dictated CRM notes about a new person) or split it when
   the messages are genuinely unrelated.
2. The watermark advances **once per run** (never per-message) as the crash barrier: a crash
   mid-run leaves the whole run "new" so a retry re-groups it identically instead of duplicating
   part of it.

### 9. New-vs-old boundary in the act step

**Where:** `skills/drain/SKILL.md` Step 4.

**Spec requirements:**

1. When loading an issue's full comment history, every comment whose id is greater than the
   issue's `lastActed` marker is marked **NEW since you last acted**; everything at or before
   `lastActed` is prior context.
2. Rationale to preserve in the prose: the drain has no memory between cycles — the thread is
   its memory — so this boundary lets it reason over the whole conversation while acting only on
   what is actually new.

### 10. Auto wiki-entity linkage (`## Meta` / `wiki_ref`)

**Where:** `skills/drain/SKILL.md` Steps 2 and 4; `references/drain-state.md`.

**Spec requirements:**

1. A drain-authored `## Meta` block lives in every issue description — seeded at creation with
   `discord_thread`, patched **in place** (never duplicated).
2. When a Step 4 route resolves a wiki page (`crm` → `people/<kebab-first-last>.md`, `capture` →
   the issue's `meetings/` page, `wiki-brain:ingest` → whatever it filed), the drain appends a
   `wiki_ref` line to the `## Meta` block.
3. On later cycles each `wiki_ref` is loaded as additional context — but only after
   **vault-validation**: the path must resolve under `{{VAULT_PATH}}` and is **rejected**
   (skip, never read) if it contains `..`, an absolute path, a drive letter, or a UNC prefix.
4. Untrusted `#inbox` message bodies are sanitized for literal `## Meta` lines (same treatment
   as the existing `## Draft` sanitizer) before being copied into a description — a message body
   can never forge the marker.
5. Wiki context **enriches; it is never a hard dependency** — a missing or moved page is skipped
   silently.

### 11. One-shot ~50-comment length nudge

**Where:** `skills/drain/SKILL.md` (Step 4b), `references/drain-state.md`.

**Spec requirements:**

1. Once an issue's comment count reaches ~50, the drain posts a single "want to start a fresh
   ticket?" suggestion to its Discord thread and sets `lengthNudged: true` on the issue's
   drain-state entry so it never repeats.
2. No compaction, no label change, no lifecycle effect — purely a tidiness nudge.

### 12. Golden trace

Upstream added `evals/golden/drain-dictation-context.md`: a 3-message dictation burst
end-to-end (settle holds mid-dictation → grouped into one issue → CRM page → `wiki_ref`
write-back → a later reply presented as NEW-since-`lastActed` with the CRM page loaded as
vault-validated context). Recreate or adapt it as this repo's golden if goldens are kept here.

**Scrub note (whole v0.9.0 section):** upstream prose says "Lucas-authored" and hard-codes the
vault as `D:\Documents\wiki-brain` — in this substrate those are the owner anchor and
`{{VAULT_PATH}}` respectively. The confirm author-gate env var is `OWNER_USER_ID`.

**Known limitation (declared upstream):** settle timing, burst grouping, `## Meta`/`wiki_ref`
write-back, and the length nudge were verified via the Pester suite and structural review only —
no live multi-message dictation burst had been exercised against Discord/Linear at ship time.
