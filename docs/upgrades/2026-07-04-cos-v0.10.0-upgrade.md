### 14. CoS Drain execution time limit (10 → 20 min)

**Where:** `scripts/register-drain-task.ps1`.

**Why:** a busy drain cycle (many issues, long Discord threads accumulated over hours) exceeded
the original 10-minute `ExecutionTimeLimit` and Windows Task Scheduler killed it mid-run
(`SCHED_S_TASK_TERMINATED`, decimal last-result `267014`). Because the task uses
`MultipleInstances IgnoreNew`, a run that finishes early costs nothing, so a larger ceiling has no
downside — it only protects against a genuinely wedged `claude -p` running forever.

**Spec requirements:**

1. The "CoS Drain" task registers with `New-ScheduledTaskSettingsSet -ExecutionTimeLimit
   (New-TimeSpan -Minutes 20)`, not 10.
2. A comment in the script explains why (a busy cycle needs headroom; a wedged run still can't run
   forever), so a future re-tightening doesn't silently regress it.

**Acceptance:** after re-running the register script,
`(Get-ScheduledTask -TaskName 'CoS Drain').Settings.ExecutionTimeLimit` reads `PT20M`.

### 15. New script: `grant-batch-logon-right.ps1`

**Where:** NEW `scripts/grant-batch-logon-right.ps1`.

**Why:** the S4U scheduled tasks (`register-drain-task.ps1` and `register-brief-task.ps1` both use
S4U specifically so no PowerShell window pops up every cycle) require the "Log on as a batch job"
right (`SeBatchLogonRight`). It is not always granted automatically at task-registration time. When
absent, every run fails at the very start with "The operator or administrator has refused the
request" (HRESULT `0x800710E0` / decimal `-2147020576`) — this looks like a credentials or
permissions bug but is actually a missing Windows local-security-policy grant, and a busy drain can
silently stop processing entirely with no other symptom.

**Spec requirements:**

1. New idempotent, **elevated-only** script grants `SeBatchLogonRight` to the current user via
   `secedit /export` → merge → `secedit /configure /areas USER_RIGHTS`.
2. The merge is **additive only** — read the existing `SeBatchLogonRight` line and append the
   current user's SID; never overwrite the line outright (that would silently strip Administrators /
   IIS_IUSRS / other existing grantees from the right).
3. It **checks `secedit`'s native exit codes explicitly** (a native-exe failure is not caught by
   `$ErrorActionPreference`), uses **unique per-run temp files with `/overwrite`** to avoid merging
   stale state, and **matches existing grantees by token** (SID), not by substring.
4. No-ops cleanly (message + exit 0) if the right is already granted — safe to re-run.
5. Documented as a standard one-time setup step alongside `register-drain-task.ps1` and
   `register-brief-task.ps1` (run once, elevated, on any new machine).

**Acceptance:** re-running the script after granting reports "already has … nothing to do"; a
subsequent CoS Drain run's `Last Result` is `0`, not `-2147020576`.

### 16. Known limitation: a confirmed calendar write can still need a live authorization

**Where:** operational — Claude Code runtime behavior, not a skill-file change. Worth a forward
pointer from `references/google-auth.md` or the drain SKILL's "propose, don't act" section so
participants aren't blindsided.

**What we learned.** The drain's own `confirmed-by-agent` gate (set only after a genuine bare-token
Discord confirm from the owner) is the plugin's correct and sufficient **internal** authorization.
It is not the only approval layer: Claude Code runs its own runtime safety check on consequential
external actions (an email or calendar invite reaching a real person), separate from the
`permissions.allow` / `deny` Bash rules in `settings.json`. A narrowly-scoped permission rule for
the Calendar events-write endpoint (e.g. `"Bash(curl *
https://www.googleapis.com/calendar/v3/calendars/*)"` — deliberately narrower than a blanket
`"Bash(curl *)"`) is correct hygiene, but did not by itself guarantee a fully silent unattended
send in testing.

**Practical implication.** A confirmed scheduling write may still need a live, explicit, same-turn
instruction to actually go out — at minimum the first time, possibly every time a send happens from
an attended interactive session. This is a Claude Code product safety behavior, not a chief-of-staff
bug, and there is no known settings-only way to fully suppress it as of this writing. Whether a
genuinely unattended cron run is treated differently is still open.

**Recommendation for the workshop.** Tell participants this is expected. The `confirmed-by-agent`
label stays the correct and sufficient authorization *within the plugin's own logic* — don't weaken
or route around it. Add the scoped calendar-endpoint permission rule regardless (minimal blast
radius, correct hygiene), but set the expectation that it's necessary, not sufficient, for a fully
silent unattended send. Don't spend workshop time trying to force full automation around this
specific gate.

**No acceptance check** — a documented limitation, not an implementation requirement.

### 17. Optional `### Interests` daily-brief module (source-linked news beats)

**Where:** `skills/briefing/SKILL.md`; NEW `skills/briefing/references/interest-beats.md`; NEW
`skills/briefing/scripts/collect_updates.py`.

**Why:** the daily brief can now carry a small, reliable news section — 3–5 **source-linked**
bullets per standing interest (topics, companies, markets, policy, sport, product niches). Adapted
from the `daily-interest-brief` skill (github.com/Nicegarrry/claude-skills) for the headless cron
path. It is **strictly additive and off by default**: zero new behavior until the participant opts
in by creating an interest list, so it cannot regress an existing brief.

**Spec requirements:**

1. **Gated on a config file that lives in the vault, not hardcoded topics.** Read the standing
   interest list from `{{VAULT_PATH}}\meta\chief-of-staff\interests.md` (one interest per bullet,
   optional `— angle` and `— feed:<url>` hints, optional `[priority]` marker). **If the file is
   absent, the module is off** — omit the `### Interests` section entirely; do not guess interests,
   do not fabricate a section.
2. **Placement:** when active, the `### Interests` section sits **after `### CRM — Meeting Prep`**
   and **before `### Proposed Focus`** in the daily brief. Cap at the **top 3 interests** by default
   (prefer any marked `[priority]`) to keep the cron fast and the brief skimmable.
3. **Candidate sourcing is deterministic-first and headless-safe.** A stdlib-only helper
   (`collect_updates.py` — no third-party deps, runs under `claude -p`) pulls RSS / Google-News
   candidates; fall back to web search only when feeds are unreachable. The model is used for source
   choice, relevance ranking, dedup, and synthesis — **never to invent** facts, dates, scores,
   quotes, or images. Default window is the last 24 hours.
4. **Degrades gracefully and yields under pressure.** If the file is absent, or every source is
   unreachable, omit the section rather than emit a thin or fabricated one. It is **read-only** (no
   vault / Linear / Google writes), respects the 2000-char Discord cap, and is the **first section
   trimmed** when a brief runs long.

**Acceptance:** with no `interests.md` present, the brief is byte-for-byte unchanged (no
`### Interests` heading). After creating `interests.md` with 1–2 interests, the next brief carries an
`### Interests` section of source-linked bullets placed between Meeting Prep and Proposed Focus, and
a deliberately unreachable feed URL degrades to "omit" rather than an error or a made-up bullet.

**Scrub notes:** the interest-list path is the participant's own vault — `{{VAULT_PATH}}` is
`D:\Documents\wiki-brain` in the reference owner's setup; each participant substitutes their own
vault root. No owner topics ship in the package; the list is the participant's to author.

### 18. Intake: media-link fast path (YouTube / video URL → ingest)

**Where:** `skills/intake/SKILL.md`.

**Why:** the intake skill previously only handled **images**. A dropped YouTube / video URL now
takes a **media-link fast path** — intake recognizes the link and *routes* it to an ingest step
rather than trying to "view" it. Division of labour: **intake routes, the knowledge-base skill
ingests** — no duplicated transcript/summarize logic.

**Spec requirements:**

1. Intake's trigger/description covers a **media link**, not just an image ("ingest this video",
   "save this", a dropped `youtube.com/watch` · `youtu.be/` · `youtube.com/shorts/` URL), and the
   `#inbox` drain may hand over a media-link item as well as an image item.
2. A **media-link fast path runs before the image flow**: if the input is a video URL, **skip the
   view/transcribe steps** (nothing to view), confirm intent (a dropped video URL is almost always
   "ingest this"), then **propose** the ingest and, on the owner's yes, hand off to the ingest step.
   Non-YouTube links are treated as normal URL sources and routed the same way. Everything else in
   the skill still applies to **images**.

**Applicability caveat (important for this repo's substrate).** The upstream hand-off target is the
`wiki-brain` knowledge-base plugin, which is **not part of the workshop chief-of-staff substrate**.
If a participant did **not** also build/install a wiki-brain-style ingest skill, item 18 has no
ingest target — apply it only as a **forward hook** (recognize the link, tell the owner "this is a
video to file — I don't have a knowledge base wired up yet"), or skip it until they add one. Item 17
has **no such dependency** and applies to every chief-of-staff.

**Acceptance:** dropping a YouTube URL into intake proposes an ingest hand-off (or, without a
knowledge base, the forward-hook message) instead of attempting to view the link as an image.
